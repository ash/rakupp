## First: 100% doesn't exist

We measured Rakudo itself on the raw files: **~96.9%**. Roast is aspirational in places ‚Äî it contains tests marked `#?rakudo todo` that the reference implementation cannot pass, tests for `::=` binding that's NYI in Rakudo, timing-sensitive S17 files that flicker even on MoarVM. So the true ceiling is roughly **97%**, and the honest phrasing of the goal is *"pass everything Rakudo-moar passes."* Against the full ~206k denominator we're at ~74%, so the real gap to the ceiling is ~23 points ‚âà ~47k assertions.

## What's just labor (most of the gap, no wall)

Judging by how the last 30 points fell, I'd say **half to two-thirds of the remaining gap is grind, not walls**:

- **One-construct parse aborts on big files** ‚Äî `Rat.new` gates 869 declared tests, postfix `i` gates 557, one `[\(‚Ä¶)]` gates 580. These are days, not months.
- **The `.new` batch** ‚Äî `Version`, `Duration`, `SetHash`, `buf8`, shaped arrays. Plumbing.
- **252 partials losing 1‚Äì2 tests each** ‚Äî pure long tail.
- **Exception fidelity** ‚Äî a lot of roast asserts the *exact* `X::` type and `.message` text of errors. That's not a language feature; it's matching Rakudo's error strings one at a time. Unbounded but trivial per item.

## The real walls (where the architecture gets tested)

Ranked by how much I'd respect them:

1. **The full grammar engine.** S05 is the largest synopsis. Our `Regex.cpp` is a capable custom engine, but real Raku regexes are *grammars*: protoregexes with **Longest Token Matching**, action classes, `<~~>` recursion, dynamic subrule dispatch, `:ratchet` semantics interacting with backtracking control. LTM in particular isn't a feature you bolt on ‚Äî it's a compilation strategy (build NFAs over all candidate tokens, run them in parallel). This is the single biggest block of unreachable tests and needs genuine design work, not patches.

2. **MOP depth.** We have enough metamodel to introspect, but roast tests parametric roles, `augment`, `.^add_method` at runtime, custom HOWs, role conflict resolution corner cases. Raku's object system is defined *by* its MOP; ours is defined by C++ classes with a MOP-shaped surface. Every layer deeper costs a real refactor.

3. **Compile-time execution.** In Raku, `BEGIN` runs *during parsing*, `constant` folds at parse time, `use` imports affect the grammar mid-file, custom operators become parseable the moment they're declared. We fake several of these convincingly, but the honest statement is: Raku is defined as a language whose compiler runs the language, and a tree-walker with a separate parse phase can only approximate that. Some tests can tell the difference.

4. **Real concurrency.** The GIL runtime passes a surprising amount of S17, but stress files (`many-processes`, fd-limit tests, scheduler timing) test *actual parallelism* and OS-level process management. Passing them means true multithreading ‚Äî the deepest surgery on the interpreter core. Even then, some of these are flaky on Rakudo; this class has a permanent noise floor (our two files that flip 400‚Üî402 with load are the small preview).

5. **Resumable exceptions.** `.resume` means an exception handler can continue execution *at the throw point* ‚Äî a continuation, essentially. It's currently our only construct that literally wedges the harness (`try.t`). A tree-walker can implement it (re-enter the eval stack), but it cuts across every `throw` path.

## The shape of the curve

The first 80% came from big buckets ‚Äî fudge support was +19 files in one change, Unicode data was +4k assertions. The remaining buckets get monotonically smaller: after the `Rat.new`-class wins land, we're into territory where a day of work buys 5‚Äì20 assertions. Nothing *stops* us before ~90‚Äì92% with the current architecture ‚Äî that's reachable by grind plus the parse-blocker list in ROAST-GAPS item 8. The band from ~92% to the ~97% ceiling is where the four walls above live, and each one is a project, not a task.

So: no mystery, no single dragon. One fake wall (100% isn't real), a wide moat of labor we've been crossing at a steady rate, and four real walls ‚Äî grammars/LTM being the tallest ‚Äî that would each deserve a design document before an implementation.
