# v2.0.0 campaign plan — the ecosystem: run real zef modules

v1.0.0 proved the *language* (90% of declared Roast, 84.5% of the corpus
byte-identical). The thing that makes an implementation *useful* is the next
layer: **programs that `use` ecosystem modules installed by zef**. That is the
v2.0.0 goal.

## Goal (measurable, like the 90% line)

> **rakupp runs the top-N ecosystem modules**, tiered:
> - **Tier 1: the module's own test suite passes ≥95%** under rakupp,
>   differentially checked against Rakudo running the same suite;
> - **Tier 2: loads + a curated usage script matches Rakudo** byte-for-byte;
> - **Tier 3: `use` loads clean** and exports resolve.
>
> **N is deliberately open**: we start with the top 50 by reverse-dependency
> count as the working set, and fix the final N (and the tier boundaries)
> after the first baseline shows how the terrain actually slopes — 50 may
> well be too small.
>
> Stretch flagship: **`zef` itself runs under rakupp** — rakupp installing its
> own modules is the 2.0 headline demo.

The metric table lives in a new `docs/MODULES.md` (the ROAST.md of this
campaign) and is generated by a `tools/run-modules.raku` harness.

## Where we already are (measured 2026-07-22)

- rakupp **already reads the zef/CURI store**: `~/.raku/short/<sha1(name)>` →
  `sources/<sha>` (Interpreter::loadModule step 2), plus `-I`/`RAKULIB`/CWD
  source trees. **All 62 dists installed on the dev machine `use` cleanly**
  (bare load, 20 s budget) — including Cro::Core and zef itself.
- What "load" doesn't yet cover: `:ver`/`:auth`/`:api` selection (we take the
  first `short/` entry), multi-version disambiguation, `%?RESOURCES`,
  `CompUnit::*` introspection beyond stubs, `EXPORT` sub semantics beyond the
  common cases, and precompilation (we always compile source — fine at our
  startup speed; see Performance below).

## Phase 0 — pick the N and pin them (selection + safety)

**Popularity metric.** raku.land shows no download counts. The honest,
computable ranking is **reverse-dependency count** over the fez + legacy
ecosystem META indexes (ugexe/Perl6-ecosystems JSON + 360.zef.pm): a module
that many dists depend on is load-bearing by definition. Cross-check the top
of the list against raku.land's "recent" pages for liveness. Expected top
cluster (to be measured, not assumed): JSON::Fast, URI, File::Temp,
File::Find, Test-adjacent utilities, Terminal::ANSIColor, JSON::(Un)Marshal,
YAMLish, Getopt::Long, License::SPDX, Digest family, Base64, DateTime
formatters, OO::Monitors, Cro's foundations.

**Freshness window.** Keep dists with a release in the last ~4 years OR a
reverse-dep count high enough that staleness is itself the finding. "Not too
fresh": pin versions ≥60 days old so we test settled releases, not
yesterday's push.

**Safety model — this is third-party code; treat it as hostile until read.**
1. **Pin, vendor, review.** A new repo `ash/raku-module-battery` vendors the
   pinned source snapshot of each selected dist (like raku-corpus vendors
   programs). Before a dist enters the battery, its sources get a human
   skim + a mechanical scan (`qx`/`run`/`shell`/socket/`unlink` inventory —
   rakupp's own `--lint` plus grep). The battery is what we run — never
   live-fetched code.
2. **Sandboxed execution.** All module test runs happen inside a sandbox:
   on macOS `sandbox-exec` with a profile denying network and confining
   writes to the work dir; in CI, a Linux container with `--network none`.
   The corpus harness protocol (sandbox copy, /dev/null stdin, alarm) is
   inherited. No battery code runs unsandboxed on the dev machine.
3. **Rakudo runs it too.** Every suite is executed under Rakudo in the same
   sandbox first — that both produces the reference verdict per test file
   and means we never run anything Rakudo-CI-grade projects haven't.

## Phase 1 — resolution correctness (find the right module)

- `:ver<…>`/`:auth<…>`/`:api<…>` adverbs on `use`/`need`/`require`: parse
  (partially done), then honor during `short/` lookup — read all entries,
  parse their dist metadata, apply Raku's version-selection rules (highest
  matching version; `+`-ranges; auth exact).
- Multi-repo chain in Rakudo's order (home, site, vendor, core paths;
  `RAKULIB`/`-I` first) + `use lib`.
- `%?RESOURCES` (dists ship data files; the store maps them via the dist
  JSON) — needed by real modules early (License::SPDX, DateTime::TimeZone).
- `require ::($name)` runtime loading; `need` without import.
- Gate: a resolution test battery (t/regression style) + no Roast/corpus
  regressions.

## Phase 2 — Tier 3→2 sweep (top 50 load, top 25 usage-verified)

The campaign engine, same loop as the 90% push: run the battery →
cluster failures → fix rakupp (or file module bugs — some WILL be module
bugs) → re-gate (Roast + corpus + battery). Expected clusters, from what
module code leans on: `EXPORT`/selective import tags, phasers at module
scope, `constant` exports, stubbed classes across files, `also is`,
delegation `handles`, `trusts`, custom metamodel bits (OO::Monitors needs
the MOP — may tier down), role parameterization depth, heredoc/pod edge
parsing in real files.

Each Tier-2 module gets a **usage script** (10–30 lines exercising its
advertised API) whose Rakudo output is the pinned reference — these live in
the battery repo like corpus `expected/`.

## Phase 3 — Tier 1: test suites (the hard 10)

Run each top-10 dist's own `t/*.t`/`t/*.rakutest` under both engines,
compare TAP (normalized). This exercises Test-module fidelity (plan
mismatches, subtests, todo/skip semantics) as much as the modules. Verdict
grid per file, aggregated per module; ≥95% pass per Tier-1 module is the
2.0 line.

## Phase 4 — NativeCall (gate for the C-binding half of the ecosystem)

A large share of high-value dists (DBIish, Digest::* fast paths, LibXML,
Readline) are NativeCall bindings. rakupp already handles `is native` for
its own compiled mode; the v2 milestone is **interpreter-mode dlopen/dlsym
FFI** for the common signature shapes (int/num/str/pointer/CArray/struct by
pointer), with libffi as the likely engine. Explicitly time-boxed: if it
threatens the schedule, v2.0.0 ships with NativeCall modules documented as
Tier-N/A and the FFI lands in 2.1.

## Performance — an explicit campaign goal, not a note

**rakupp's speed must not go down** as module machinery lands. Two distinct
commitments:

1. **No regressions on what we have.** The kernel/opt benchmarks and the
   YAMLish real-world figure join the per-leg gates for this campaign
   (`tools/perf-guard.raku` / quick interleaved A/B on touched paths) — the
   90% campaign's ~8% YAMLish accretion taught us this cost creeps in
   silently. Resolution-path additions (version selection, dist-JSON reads)
   must stay off the hot path: resolve once per `use`, cache per process.
2. **Module load time is a first-class metric.** The battery harness records
   per-module load and suite wall-clock under both engines from day one;
   MODULES.md publishes them. Grammar-heavy dists (Cro) compile from source
   on every run — our fast parse may well beat Rakudo's precomp-cache *hit*,
   and certainly its cold path; measure, don't assume.

**Our own precomp** (a source-hash-keyed serialized-AST cache per compunit,
Rakudo-precomp-like in role but format-ours) is a *considered* item: worth
building only if the day-one measurements say module load is a real pain at
battery scale. If built, it doubles as the seed of incremental tooling. The
decision point is after the Tier-3 baseline — with numbers, not vibes.

## Working agreements (inherited from the 90% campaign)

- Zero-regression gates: full Roast + corpus differential + battery, per leg.
- Every gate-caught or user-reported bug gets a `t/regression/` case.
- Divergences vs Rakudo logged (spec-divergences memory / CORPUS-DIFF style
  triage docs). "The module itself is buggy / Rakudo-illegal" is a finding
  too — recorded in **our own triage files only**
  ([MODULE-FINDINGS.md](MODULE-FINDINGS.md), here in docs/dev). **We do not submit bug reports to module authors**;
  everything stays in-repo, for us.
- Docs numbers synced per leg: MODULES.md joins ROAST.md/FEATURES.md in the
  doc-sync discipline.
